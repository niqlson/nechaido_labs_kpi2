# Вимоги до Online Shop

## Функціональні вимоги (Functional Requirements)

### Управління користувачами та доступом
FR1. Система має надавати можливість реєстрації нового облікового запису
FR2. Система має надавати можливість авторизації в системі
FR3. Система має надавати можливість безпечного виходу з системи
FR4. Система має розрізняти ролі: Гість, Користувач, Адміністратор

### Перегляд та пошук товарів
FR5. Система має надавати можливість перегляду каталогу товарів
FR6. Система має надавати функціонал пошуку товарів за ключовими словами
FR7. Система має надавати детальну інформацію про товар
FR8. Система має підтримувати фільтрацію товарів за категоріями

### Управління кошиком
FR9. Система має надавати можливість додавання товарів до кошика
FR10. Система має надавати можливість зміни кількості товару в кошику
FR11. Система має надавати можливість видалення товарів з кошика
FR12. Система має відображати актуальний стан кошика

### Процес оформлення замовлення
FR13. Система має надавати процес оформлення замовлення
FR14. Система має надавати можливість введення даних доставки
FR15. Система має надавати вибір способу оплати
FR16. Система має підтримувати застосування промокодів та знижок
FR17. Система має інтегруватися з платіжними сервісами
FR18. Система має підтверджувати успішне оформлення замовлення

### Управління замовленнями
FR19. Система має надавати можливість відстеження статусу замовлення
FR20. Система має надавати історію замовлень для користувачів
FR21. Система має надавати адміністраторам перегляд всіх замовлень

### Відгуки та рейтинги
FR22. Система має надавати можливість написання відгуків про товари
FR23. Система має надавати можливість редагування власних відгуків
FR24. Система має надавати можливість видалення власних відгуків

### Система бронювання
FR25. Система має надавати можливість бронювання товарів
FR26. Система має надавати можливість скасування бронювань
FR27. Система має надавати перегляд статусу бронювань
FR28. Система має надавати можливість підтвердження отримання заброньованих товарів
FR29. Система має надавати адміністраторам модерацію бронювань

### Адміністративні функції
FR30. Система має надавати управління товарами (додавання, редагування, видалення)
FR31. Система має надавати управління складськими запасами
FR32. Система має надавати управління системами знижок
FR33. Система має надавати управління бронюваннями

## Нефункціональні вимоги, що випливають з Use Case діаграми

### Безпека
**NFR1. Захист автентифікації**
- Use Case: `Login`, `Register Account`
- Вимога: Система має використовувати хешування паролів алгоритмом SHA-256
- Обґрунтування: Критично для захисту облікових записів користувачів

**NFR2. Безпека платіжних операцій**  
- Use Case: `Confirm Payment`
- Вимога: Всі транзакції мають використовувати HTTPS та відповідати PCI DSS
- Обґрунтування: Конфіденційність платіжних даних є обов'язковою

**NFR3. Захист від несанкціонованого доступу**
- Use Case: `Manage Products`, `Manage Discounts`, `Admin Reservations`
- Вимога: Доступ до адміністративних функцій лише для актора `Admin`
- Обґрунтування: Розділення прав доступу між ролями

### Надійність
**NFR4. Цілісність транзакцій замовлень**
- Use Case: `Place Order`, `Checkout`
- Вимога: Система має гарантувати, що замовлення або повністю зберігається, або повністю скасовується
- Обґрунтування: Уникнення втрати даних при збоях

**NFR5. Консистентність даних кошика**
- Use Case: `Add to Cart`, `Change Item Quantity`, `Remove from Cart`
- Вимога: Зміни в кошику мають негайно відображатися для користувача
- Обґрунтування: Критично для користувацького досвіду

### Зручність використання
**NFR6. Інтуїтивність процесу оформлення**
- Use Case: `Checkout` (включає 5 під-сценаріїв)
- Вимога: Процес має мати лінійний потік з чіткими кроками
- Обґрунтування: Зменшення кількості покинутих кошиків

**NFR7. Простота управління контентом**
- Use Case: `Manage Reviews`, `Manage Reservations`
- Вимога: Користувачі мають мати змогу легко знаходити свої відгуки/бронювання
- Обґрунтування: Покращення користувацького досвіду

*Примітка: Об'єктивно показати коректність виконання нефункціональних вимог використовуючи тільки use case діаграму не можно, тому ці вимоги були написанні зі загальних міркувань.*
